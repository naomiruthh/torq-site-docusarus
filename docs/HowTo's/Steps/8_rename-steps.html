The step name plays two important roles in workflows:

  • It's the step's unique identifier on the designer.
  • It enables you to access and reference the step elsewhere in the workflow via JSON expression. For example, {{ $.<step_name>.results }}

When you change the name of a step, the step references are automatically updated. 

Duplicate step names

If there are two or more steps with the same name and you want to rename one of them, you have to decide whether to change the step name in any existing references or keep the references as they are, despite the step name change.

The following example includes a conclusion string that gets the information from one of the steps named RF Result. Only one of the steps with the duplicate name is executed according to the condition evaluation. 

Rename a step and change the references 

Rename one of the steps with the shared name and make sure the Update all references checkbox is checked to change the step name in the references. The steps that reference the step you're renaming are listed for your convenience.

The result is that only the step with the new name is now being referenced. The other steps with the shared name are no longer referenced anywhere.

The step is renamed and the reference is changed.

Rename a step and keep the current references

Rename one of the steps with the shared name and uncheck the Update all references checkbox to keep the references as they are. The steps that reference the step you're renaming are listed for your convenience.

The result is that the step with the new name isn't referenced anywhere. The steps with the shared name will continue to be referenced. You can change the references manually as required.

The step is renamed, but the reference isn't changed.
